# Phase 03 — Playground (Worker, Task Modes, Tests UI)

**Prerequisites:** Phase [`02.md`](02.md) complete.  
**Next phase:** [`04.md`](04.md) Content expansion and polish.

**Context:** Learners switch to **Practice** to write code, click **Run**, see captured **console** output, and get **pass/fail** results. Execution must stay in the **browser**; use a **Web Worker** with a **timeout** and worker termination to mitigate infinite loops. Early topics must not require `return` before functions are taught—use **binding** and **console** task modes.

---

## Objectives

1. **Playground runner:** post message to worker with task id + user code + task spec; receive logs, test results, errors, duration.
2. **Task modes:**
   - **bindings:** user code runs in a scope where expected variables are asserted (e.g. `name`, `age` after assignments).
   - **console:** compare captured `console.log` lines to expected (order-sensitive or normalized per spec).
   - **function:** user defines a named function; runner invokes with test `args`, compares to `expected` (deep equality helper).
3. **UI:** Code editor (textarea minimum; Monaco optional later), Run, Reset, problem statement (description, input, expected), **Test results** list, **Hints** (collapsible ladder).
4. **Safety:** worker timeout (e.g. 1500ms configurable per task); on timeout, show friendly message and kill worker.
5. **Integration:** Per-topic task list in `src/data/playground/*.ts`; selecting task loads starter code and instructions.

## Tasks

### 3.1 Types

- [ ] `src/types/playground.ts`: `PlaygroundTask`, `TaskMode`, `BindingExpectation`, `ConsoleExpectation`, `FunctionTestCase`, `PlaygroundResult`.
- [ ] Task metadata: `id`, `topicId`, `title`, `description`, `starterCode`, `hints[]`, `mode`, mode-specific fields, `timeoutMs?`.

### 3.2 Worker

- [ ] `src/workers/playground.worker.ts`: receive run message, create appropriate wrapper:
  - **bindings:** `new Function('console', '"use strict";\n' + userCode + '\nreturn { ... };')` with explicit list of binding names from task.
  - **console:** wrap `console.log`, collect strings; then run user code.
  - **function:** append or expect `function name(...) { ... }` and call `globalThis[name](...args)`.
- [ ] Handle thrown errors; stringify stack in dev-only or short message in UI.
- [ ] `src/engine/playgroundRunner.ts` (main thread): `Worker` lifecycle, `URL.createObjectURL` or Vite worker import, terminate on timeout.

### 3.3 Assertions

- [ ] `src/engine/assertions.ts`: `deepEqual` for JSON-safe values; number tolerance optional; order of array comparison for console mode.

### 3.4 UI components

- [ ] `PlaygroundView`: layout = problem + editor + actions + console + tests.
- [ ] `TaskPicker` or pills per topic.
- [ ] `TestResults`: per-case pass/fail, expected vs received.
- [ ] `HintPanel`: reveal hints one by one or expand all.
- [ ] Reuse **ConsolePanel** styling or shared `LogOutput` component for learner logs.

### 3.5 State and hooks

- [ ] `hooks/usePlaygroundRunner.ts`: idle / running / success / error; debounce Run; clear on task change.

### 3.6 Initial tasks (minimal vertical slice)

- [ ] **Variables topic:** 1–2 `bindings` tasks (e.g. profile fields, cart total variable).
- [ ] **Loops topic:** 1 `function` task (e.g. sum savings over array of objects) *after* functions are introduced in curriculum order—or scope to topic order: if loops come after functions in app, OK.
- [ ] **Console topic or operators:** 1 `console` task (e.g. print final price string).

### 3.7 Settings (optional in this phase)

- [ ] Persist last selected task per topic in `localStorage` (nice-to-have).

## Acceptance Criteria

| # | Criterion |
|---|-----------|
| A1 | **Practice** tab is no longer placeholder: user can select a task, edit code, click Run. |
| A2 | `console.log` from user code appears in the educational log area, **not** requiring browser DevTools. |
| A3 | **bindings** task passes only when declared variables match expected values (types as specified). |
| A4 | **console** task passes when log output matches spec. |
| A5 | **function** task passes when all test cases match. |
| A6 | Long-running / infinite loops are stopped by timeout; UI shows clear message; worker terminated. |
| A7 | Syntax/runtime errors show user-friendly summary + expandable detail. |
| A8 | At least **3 tasks** across **≥2 modes** prove the pipeline (e.g. bindings + function + console). |
| A9 | No `eval` in main thread for user code; execution isolated to worker path as designed. |
