# Phase 04 — Curriculum Expansion & Polish

**Prerequisites:** Phases [`01.md`](01.md)–[`03.md`](03.md) complete.  
**Next phase:** None (maintenance / future enhancements).

**Context:** The platform exists: guided examples (Phase 02) + playground (Phase 03). This phase fills **content**, aligns **learning order**, improves **UX/accessibility/responsiveness**, and retires reliance on [`frontend/init_example.html`](../../frontend/init_example.html) as documentation of behavior.

---

## Objectives

1. **Coverage:** Target **14 topics** (see [`overview.md`](overview.md) table). Each topic ships with **≥2 guided examples** and **≥2 playground tasks** unless explicitly deferred (DOM/modules may phase in mock-heavy tasks).
2. **Pedagogy:** Every guided example ties to real-world copy; loop examples expose **init / condition / body / update / exit** in steps.
3. **UX polish:** Mobile layout (stack panels, sticky controls), keyboard nav, focus states, reduced-motion support, persisted settings (sound, speed, theme if added).
4. **Legacy:** README points to `bun run dev`; `init_example.html` labeled legacy or removed only if redundant (team choice).

---

## Tasks

### 4.1 Content — guided examples

- [ ] For each topic in roadmap: write 2–4 examples using established types (`CodeLine` segments, `ExecutionStep`).
- [ ] Priority order if incremental delivery: Variables → Operators → Conditionals → Functions → Arrays → Objects → Loops → Classes → Modern JS → Async → Errors → DOM → Events → Modules.
- [ ] Include **for-loop** example: customers + **total savings** (or invoicing totals) with segment highlights on condition vs `i++`.

### 4.2 Content — playground

- [ ] For each topic: 2–4 tasks with correct **mode** (variables → bindings/console; post-functions → function mode).
- [ ] Align task difficulty with prerequisite topics (no `map` before arrays topic, etc.).
- [ ] Hints: at least two levels per harder task.

### 4.3 Topic intros

- [ ] UI block per topic: **What it is**, **Why it matters**, **Real-world examples** (bullets)—as in original spec.

### 4.4 Responsiveness & a11y

- [ ] Breakpoints: sidebar becomes tabs or drawer; code/editor gets minimum height.
- [ ] Visible focus rings; aria labels on icon buttons (play, sound).
- [ ] `prefers-reduced-motion`: reduce/disable confetti, shorten transitions.

### 4.5 Settings persistence

- [ ] localStorage: sound on/off, default speed, last topic/example/task.

### 4.6 Quality bar

- [ ] Smoke pass: each topic opens, first example plays to end without step errors.
- [ ] Smoke pass: sample playground tasks per topic pass with reference solutions (store private `solution` snippets in dev-only or internal doc—not shipped to learner if undesired).

### 4.7 Documentation

- [ ] Root or `frontend/README.md`: setup, scripts, architecture pointer to `docs/codestep/`.
- [ ] Optional: Contributing guide for adding a new guided example (`DATA.md` template).

## Acceptance Criteria

| # | Criterion |
|---|-----------|
| A1 | All **14 topics** appear in navigation with intros and ≥2 guided examples each **or** documented exceptions (e.g. Modules with simulated editor) approved in README. |
| A2 | All topics have **≥2 playground tasks** with passing reference solutions **or** documented exceptions with workaround timeline. |
| A3 | Variables-only tasks never require `function`/`return` unless curriculum order places them after Phase 03 function introduction in app. |
| A4 | App is usable at **375px** width without horizontal breakage of core flows. |
| A5 | Keyboard shortcuts documented in-ui; focus management does not trap in editor. |
| A6 | `prefers-reduced-motion` respected for major animations. |
| A7 | New user can complete “happy path”: open app → Learn → finish one example → Practice → pass one task. |
| A8 | `init_example.html` clearly superseded by React app entry (README + optional deprecation notice in file header). |
