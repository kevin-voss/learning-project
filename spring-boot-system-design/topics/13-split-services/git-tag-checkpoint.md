# Git tag checkpoint: save the monolith before you break it

Hands-on companion to [Step 13](README.md). Do this **before** touching any code in this step. ~15 minutes, and it may be the first time you ever use Git — that's fine, every command is spelled out.

## The problem

Step 13 doesn't add a file or two; it **reshapes the whole project**. You'll move code out of `applications/parcelpilot` into two new services, change package names, split the `pom.xml` — dozens of coordinated edits. If it goes wrong halfway (it might; that's normal), you want one thing above all: **the working monolith back**, exactly as it was when every step-12 acceptance box was ticked.

"I'll remember what I changed" does not survive contact with a two-hour refactor. You need a snapshot.

## The solution: version control + a tag

Put the monolith under **Git** and pin a **tag** — a permanent, human-readable name on this exact state — before splitting anything. This is precisely what [PROJECT-STORY.md](../../PROJECT-STORY.md) asks for at this point ("preserve the final monolith as a tagged Git commit"). Background on what Git is and the handful of commands this course needs: [Git for this course](../../references/git-for-this-course.md).

## Do it, command by command

### 1. Go to the monolith and initialize a repository

```bash
cd applications/parcelpilot
git init
```

Expected output:

```text
Initialized empty Git repository in .../applications/parcelpilot/.git/
```

The hidden `.git/` folder is where all history will live. Nothing else changed.

### 2. Tell Git what to ignore

Create a file named `.gitignore` (exactly that, starting with a dot) in `applications/parcelpilot` with this content:

```gitignore
# Maven build output — regenerated by every build, never commit
target/

# IDE and editor files — personal, not part of the project
.idea/
*.iml
.vscode/

# OS noise
.DS_Store

# optional: personal scratch notes, if you keep them here
NOTES.md
```

Without this, your snapshot would include megabytes of compiled classes from `target/` that every `mvn package` regenerates anyway.

### 3. Stage and commit — the snapshot itself

```bash
git add .
git commit -m "Monolith complete through step 12"
```

Expected output (numbers will differ):

```text
[main (root-commit) 3f2a91c] Monolith complete through step 12
 27 files changed, 1450 insertions(+)
 create mode 100644 pom.xml
 create mode 100644 src/main/java/com/parcelpilot/Parcel.java
 ...
```

If Git asks you to configure your name and email first, do what it says (`git config --global user.name "..."` and `git config --global user.email "..."`), then re-run the commit.

### 4. Pin the tag

```bash
git tag v-step-12-monolith
```

No output means success. That name now points at this commit **forever** — however much you change afterwards.

### 5. Verify

```bash
git log --oneline
```

```text
3f2a91c (HEAD -> main, tag: v-step-12-monolith) Monolith complete through step 12
```

```bash
git tag
```

```text
v-step-12-monolith
```

One commit, one tag. Checkpoint done — go split some services.

## Getting back (when you need it)

You probably won't. But if the split goes sideways:

**Just look at how a file used to be** (most common case — "what did the old controller look like?"):

```bash
git show v-step-12-monolith:src/main/java/com/parcelpilot/ParcelController.java
```

**Restore all files to the checkpoint** — first stash your half-broken changes aside (they're kept, not deleted), then pull the tagged files back:

```bash
git stash
git checkout v-step-12-monolith -- .
```

Now the folder contains the working monolith again. (If you later want the stashed mess back to salvage something: `git stash pop`.)

## Pros and cons: tag checkpoint vs copy-the-folder

The README mentions the alternative — copying to `checkpoints/08-event-driven-monolith`. Both are valid:

| | Git tag | Folder copy |
|---|---|---|
| Effort | 5 commands, once | one `cp -r` |
| Disk | tiny (Git stores changes compactly) | full duplicate |
| Later payoff | `git diff` shows exactly what the split changed; a commit per finished step from now on | none — it's a frozen copy |
| Learning cost | you must meet Git today | zero |

If Git feels like too much on top of a big refactor day, the folder copy is honestly fine. But Git pays off immediately: from this point on you can commit after every green acceptance checklist and never again lose more than one step of work — the habit is described in [Git for this course](../../references/git-for-this-course.md).
